Benchmark setup and baseline definition

What A and B mean

B. Standard baseline

Cold execution before useful cache reuse. Repeated geocode, route, and AI-generation work is treated as a direct backend/external-service path.

A. Optimized backend

Warm execution with Redis cache, persisted place/geocode reuse, route prewarm, parallel route calls, and route fallback behavior.

Interpretation rule

Percentiles and measured service rate are preferred. When logs only contain averages, the dashboard marks service rate as a latency-derived proxy.

How to present it

Backend/Data narrative

The stress-test log provides the controlled A/B evidence: B is the cold baseline and A is the warm optimized path after caches, persisted geocode data, and route reuse are available.

Runtime counters show whether the same optimization holds during normal use: external API calls are compared against a standard baseline estimate, and latency savings are calculated from measured request timing rather than hand-written values.

Service rate should ideally come from measured benchmark duration and concurrency. If a log does not include those fields, this dashboard explicitly labels the value as a latency-derived proxy, not as a measured load-test service rate.
