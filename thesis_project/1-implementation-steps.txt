# 1.txt 首次生成链路实施步骤

## 实现注意事项：防止再次形成巨型 Service

这份实施计划的首要工程约束是：不能在新链路中再次形成类似旧 `PlanProcessorService` 那样的巨型文件。`PlanProcessorService` 曾经接近 7800 行，问题不只是文件长，而是 prompt、AI 调用、JSON 解析、POI 查询、排程、路线、校验、修复、缓存、日志和响应组装混在同一个类里，导致任何改动都会牵动整条链路。

后续实现必须遵守以下规则：

- 单个 application service 原则上不超过 300-500 行；超过时优先拆成明确职责的小服务。
- 单个方法原则上不超过 80-120 行；复杂流程拆成私有步骤或独立 domain/application service。
- `GenerateInitialPlanUseCase` 只做编排，不承载具体算法。
- `PlanProcessorService` 只保留 legacy facade / adapter 角色，不再往里面添加新主链路逻辑。
- Planning Agent、Zone Retrieval、Coverage、Scheduler、Validator、Repair、Route、Copy Polish 必须是独立边界。
- 新增功能时优先新增小接口和小实现，不把临时代码塞进已有大类。
- Coverage 不能调用 Scheduler 内部逻辑；Scheduler 不能自己做 Coverage Resolution。
- Agent 不能直接生成 POI；Scheduler 不能直接调用 LLM。
- 外部 API 调用必须通过 infrastructure adapter 或明确 application interface，不散落在业务流程中。
- DTO、domain model、persistence entity 不要混用；转换逻辑集中在 assembler / mapper。
- 每完成一个阶段必须补充或更新测试，至少覆盖主路径和关键降级路径。
- 新增阶段如果需要临时实现，必须在文档中标明“临时语义”和后续替换点，避免临时方案固化。
- 任何类如果开始同时承担三个以上职责，应立即拆分，而不是继续追加参数和私有方法。
- 不为“先跑通”牺牲边界：可以先做简化算法，但不能把边界做反。

推荐文件组织方式：

```text
planning/application
  GenerateInitialPlanUseCase
  LightweightRequestPreParser
  RetrievalQueryBuilder
  PlanningAgent
  PlanningSpecificationValidator
  ZoneContextBuilder

catalog/application
  DestinationResolver
  DestinationCatalog
  PlanningZoneRetrievalService
  CoverageInventoryBuilder
  CoverageGapResolutionService

planning/localfast
  LocalPlanGeneratorService
  SlotFillingScheduler
  CandidateScoringService
  DayScheduleBuilder

planning/quality
  DeterministicQualityValidator
  LocalRepairService

routing/application
  LocalRouteEstimateService
  PreciseRouteEnhancementService
```

判断一个实现是否开始变坏：

```text
一个类既解析请求，又查 POI，又调模型，又排时间，又修复计划
-> 必须拆分

一个方法需要传入超过 6-8 个业务参数
-> 应该引入 command/context/result 对象

一个测试只能通过大量 mock 一个巨型 service 才能覆盖
-> 说明边界已经过大

一个阶段的实现无法单独验证
-> 说明模块边界不清
```

## 外部服务选型与 Adapter 边界

后续外部能力的推荐选型：

```text
模型：
DeepSeek-v4-pro

地图与地点：
高德 API

天气：
和风天气
```

重要原则：
- 业务层不能直接依赖 DeepSeek、高德或和风天气的 SDK / HTTP DTO。
- 所有外部服务都必须通过 application interface + infrastructure adapter 接入。
- 外部服务返回的数据必须先标准化为系统内部 domain model，再进入 Coverage、Scheduler、Validator。
- 外部 API 不参与主链路自由决策，只提供事实数据或候选补充。
- 外部 API 失败时必须有 local-first fallback，不能导致首屏结构化计划完全不可用。
- 供应商替换不能影响 `GenerateInitialPlanUseCase`、`DestinationCatalog`、`ItineraryGenerator`、`PlanQualityService` 等核心接口。

模型接入建议：

```text
planning/application/PlanningModel.java
planning/infrastructure/deepseek/DeepSeekPlanningModelAdapter.java
```

DeepSeek-v4-pro 的职责：
- 实现真实 `Single Planning Agent`。
- 输出 constraints、specialEvents、dayStrategies。
- 输出动态重规划候选 PlanPatch。

DeepSeek-v4-pro 不负责：
- 直接生成最终 POI。
- 直接修改数据库。
- 直接决定外部 API 调用。
- 绕过 Coverage / Scheduler / Validator。

地图与地点服务映射：

| 现有服务 | 高德替代 |
| --- | --- |
| Google Places Search | 搜索 POI 2.0 |
| Google Place Details | POI 2.0 ID 搜索 |
| Google Geocoding | 地理/逆地理编码 |
| Geoapify Route | 路径规划 2.0 |
| 本地 POI Zone 判断 | 行政区查询 + 自定义 Planning Zone |
| 前端地点搜索框 | 输入提示 |
| 原始 GPS/Google 坐标 | 坐标转换 |
| 动态交通扰动 | 交通事件 + 交通态势 |
| 未来驾车 ETA | 高级路径规划，可选 |

```text
routing/application/RouteCalculationService.java

weather/application/WeatherService.java
```

建议 adapter：

```text
catalog/infrastructure/amap/AmapPoiSearchAdapter.java
catalog/infrastructure/amap/AmapPlaceDetailAdapter.java
routing/infrastructure/amap/AmapGeocodingAdapter.java
routing/infrastructure/amap/AmapRouteCalculationAdapter.java
routing/infrastructure/amap/AmapTrafficAdapter.java
routing/infrastructure/amap/AmapCoordinateTransformAdapter.java
weather/infrastructure/qweather/QWeatherAdapter.java
planning/infrastructure/deepseek/DeepSeekPlanningModelAdapter.java
```

高德 API 使用边界：
- 搜索 POI 2.0 只用于 gap-driven enrichment、地点搜索和候选扩充，不默认替代本地 catalogue。
- POI 2.0 ID 搜索用于补全详情和验证外部候选。
- 地理/逆地理编码用于坐标补全、地址校验和用户输入解析。
- 路径规划 2.0 用于异步精确路线，不阻塞首屏 local-fast。
- 行政区查询只能辅助 zone 判断，最终仍以系统自定义 Planning Zone 为准。
- 坐标转换用于处理历史 Google/GPS 坐标与高德坐标系差异。
- 交通事件和交通态势用于动态扰动和重规划，不进入普通首次生成的同步关键路径。

和风天气使用边界：
- 用于 weather snapshot、降雨/高温/恶劣天气判断和 indoor fallback。
- 首屏可以使用缓存或默认天气风险，不强制等待实时天气。
- 天气影响应进入 constraints、Coverage、Scheduler scoring 和异步 replan，不直接让模型自由改计划。

迁移时不要一次性替换所有外部服务。建议顺序：
1. 先抽象接口，保留现有实现。
2. 增加高德 geocoding / route adapter。
3. 增加高德 POI search / detail adapter。
4. 增加和风天气 adapter。
5. 增加 DeepSeek planning model adapter。
6. 每替换一个 adapter，都保留现有测试和 fallback。

本文档用于把当前 Spring Boot + MySQL 智能旅行规划系统，从当前 `local-fast / local-first` 主路径逐步迁移到 `1.txt` 定义的最终首次生成链路。

标记规则：
- 已完成的步骤在标题末尾追加 `（complete）`。
- 后续每完成一部分，就更新对应步骤并追加 `（complete）`。
- 不推翻当前可运行链路，不删除旧链路；先保证 `/api/v1/plans/draft` 的 local-fast 首屏结构化计划始终可用。
- 当前没有向量召回前，不把运行模式命名为 Hybrid RAG；先使用 `ZONE_GUIDED_LOCAL_FIRST` 或 `STRUCTURED_RETRIEVAL_LOCAL_FIRST`。

最终目标链路：

```text
Lightweight Request Pre-parser
-> Destination Resolver
-> Retrieval Query Builder
-> Structured / Hybrid Planning Zone Retrieval
-> Zone Context Builder
-> Single Planning Agent
-> Planning Specification Validator
-> Trip Skeleton Builder
-> Catalog Inventory / Coverage View
-> Minimum Catalog Coverage Check
-> Coverage Gap Resolution
-> Final Candidate Pool
-> Local-first Scheduler
-> Deterministic Quality Validator
-> Local Repair
-> 返回首屏结构化计划
-> 异步增强
```

重要约束：
- RAG 只负责 Planning Zone 召回和 Zone 能力上下文。
- Agent 只负责 constraints、specialEvents、dayStrategies，不生成具体 POI。
- Scheduler、Coverage、Validator、Repair 保持确定性后端能力。
- Hard Gap 不能长期只打日志；过渡期可以观测，最终必须触发 resolution。
- Coverage 用的是 request-scoped inventory/view，不等同于最终给 Scheduler 使用的 Candidate Pool。

当前代码基线：
- `GenerateInitialPlanUseCase` 已是首次生成主入口。
- `PlanProcessorService` 已基本退为 legacy facade / adapter。
- `PlanDraft` 已作为内部 itinerary model 使用。
- `DestinationCatalog`、`ItineraryGenerator`、`RoutePlanningService`、`DefaultPlanQualityService` 等边界已存在。
- local-fast 主路径已经基于本地 POI catalogue 生成结构化计划。
- LLM 主要保留在 AI-first legacy path 和异步 copy polish 能力中。

## 阶段 3：建立 TripSkeleton / TripSlot（complete）

目标：
- 在 Coverage Check 前先知道每天需要多少 activity、meal、special event slots。
- 对齐 `1.txt` 中 “Trip Skeleton Builder 必须在 Coverage Check 前执行” 的顺序。

已完成内容：
- 新增 `thesis.project.gu.planning.domain.TripSkeleton`。
- 新增 `thesis.project.gu.planning.domain.TripSlot`。
- `DestinationCatalog.buildTripSkeleton(...)` 已建立接口。
- `LocalPoiCatalogService.buildTripSkeleton(...)` 已提供本地默认实现。

当前 skeleton 规则：
- 根据 `days` 生成 day skeleton。
- 根据 `pace` 决定 activity slot 数：
  - relaxed: 2
  - normal: 3
  - rush/fast/intense: 4
- 有儿童时降低每日 activity 密度。
- 每天默认包含 activity slots、lunch slot、dinner slot。
- relaxed pace 默认 startTime 为 `10:00`，其他默认为 `09:00`。
- 如果存在 birthday special event，dinner slot 会带 `birthday-suitable` capability。

当前实现边界：
- `specialEvents` 目前不会自动生成，除非未来上游填入。
- 还没有复杂 day allocation。
- 还没有 rain fallback / indoor slots。
- slot capability matching 当前是最小版本。

后续可演进：
- 根据 `DayStrategy` 生成更细的 day-level slot。
- 加入 indoor/outdoor、birthday、reservation、late start 等 capability。
- 加入 half-day/full-day allocation。
- 根据 zoneType 生成不同 skeleton，例如大型景点不强制 3 个 activity POI。

## 阶段 4：建立 CoverageResult / CoverageGap，并扩展 DestinationCatalog（complete）

目标：
- Coverage Check 要区分最低可生成阈值和理想候选阈值。
- Soft Gap 可以继续生成；Hard Gap 最终必须触发 resolution，不能长期无条件继续。
- Coverage 使用的是 coverage inventory / available view，不等同于最终 Candidate Pool。

已完成内容：
- 新增 `thesis.project.gu.catalog.domain.CoverageResult`。
- 新增 `thesis.project.gu.catalog.domain.CoverageGap`。
- `DestinationCatalog` 新增：
  - `findAvailableZones(...)`
  - `buildTripSkeleton(...)`
  - `checkCoverage(...)`
  - `buildCandidatePool(...)`
- `LocalPoiCatalogService.checkCoverage(...)` 已支持：
  - activity slot coverage
  - lunch slot coverage
  - dinner slot coverage
  - hard gaps
  - soft gaps

当前判断规则：

```text
availableCandidateCount < requiredUsageCount
-> hard gap, generatable=false

requiredUsageCount <= availableCandidateCount < preferredCandidateCount
-> soft gap, generatable=true, preferredCoverageMet=false

availableCandidateCount >= preferredCandidateCount
-> sufficient
```

当前实现边界：
- hard gap 目前只记录和打日志，不阻断首屏，这是过渡状态。
- 不调用外部地点 API。
- 不做 fallback zone resolution。
- 不修改 candidate pool。
- 当前 `buildCandidatePool` 在 coverage 前被复用为数据读取手段，语义上应改成 `buildCoverageInventory` 或 `buildPreliminaryCandidateView`。

后续必须调整：
- 新增 `CatalogInventory` 或 `CoverageInventory`。
- `checkCoverage(...)` 输入从 final candidate pool 改成 request-scoped coverage view。
- Hard Gap 处理顺序：
  - fallback Zone
  - 同城可行 Zone
  - gap-driven 外部 API 补充
  - 降低非必要 slot
  - 仍不足则返回 partial/warning 或明确失败
- Soft Gap 允许缩小备用候选池并继续生成。

## 阶段 17：Coverage Gap Resolution

- 有 semanticProfile。
- 有 embeddingVersion。
- 有 vector index。
- 有 Top-K recall trace。
- 有 business rerank score。

完成后才可以把 planningMode 升级为：

```text
HYBRID_RAG_GUIDED_LOCAL_FIRST
```
